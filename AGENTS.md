## 七、工具使用策略

### AI 原生优先（最高优先级）

> **铁律：能用 AI 原生能力解决的，绝不写 Python 脚本！**

**AI 原生能力**：read_file、edit_file、grep_search、glob_search、memory_search

**禁止**：为知识库/文件管理/统计写 Python 脚本

**Python 只能用于**：操作浏览器、编译 C++、系统操作、用户明确要求

| 场景 | 首选 |
|------|------|

| 按含义搜索 | memory_search |

### 浏览器操作

严禁直接调用 browser_use 的 start/open/connect_cdp

## 十一、Heartbeats 主动维护

### 人格状态检查

默认 Planner，发现异常立即恢复

### 记忆维护（每隔几天）

**触发条件**：MEMORY.md 超过 200 行，或每日笔记超过 10 个

**梳理流程**：

```
1. 读取 MEMORY.md，分析内容结构
2. 分类整理：
   - 工具设置（保留）→ 精简，去掉过时信息
   - 经验教训（保留）→ 合并同类，删除重复
   - 项目进展（归档）→ 移到项目专属文件
   - 过时信息（删除）→ 已失效的配置、已完成的任务
3. 更新 MEMORY.md（只保留核心内容）
4. 创建归档文件 memory/archive/YYYY-MM.md（存放移出的内容）
```

**MEMORY.md 标准结构**（不超过 150 行）：

```markdown
# 长期记忆

## 一、工具设置（核心配置）
- 文件保存偏好
- AI 原生优先原则
- OJ 系统信息
- 浏览器自动化流程
- CDN 源偏好

## 二、用户偏好（工作习惯）
- 讲义目录结构
- Git 提交策略
- 编码风格

## 三、血泪教训（重要！）
- config.yaml 格式错误
- 不读 SKILL.md 就动手

## 四、重要承诺
- 哥哥的信任（2026-03-14）
```

**应该移出的内容**：

| 内容类型 | 移出位置 | 原因 |
|---------|---------|------|
| 项目进展详情 | `memory/archive/项目-YYYY-MM.md` | 只需要当前状态，不需要历史 |
| 考试记录 | `memory/archive/考试记录.md` | 已沉淀到 SOUL.md |
| 技能配置详情 | `skills/xxx/SKILL.md` | 技能自己管理 |
| 临时任务记录 | `memory/YYYY-MM-DD.md` | 每日笔记已有 |

### 每日笔记清理（每周）

**触发条件**：memory/ 目录下超过 20 个日期文件

**清理流程**：
1. 保留最近 7 天的笔记
2. 7-30 天前的笔记：提炼有价值内容到 MEMORY.md，然后删除
3. 30 天前的笔记：归档到 `memory/archive/YYYY-MM.md`，删除原文件

### 待办任务维护（每次 heartbeat）

**pending_tasks.md 维护**：

```
检查规则：
1. 已完成的任务 → 移到"已完成 ✅"区，保留最近 10 条
2. 阻塞超过 7 天的任务 → 标记"⚠️ 长期阻塞"，询问用户是否放弃
3. 已完成超过 30 天 → 从"已完成"区删除，归档到 archive
```

### 已解决问题维护（每次 heartbeat）

**resolved_questions.md 维护**：

```
检查规则：
1. 超过 30 天未引用的问题 → 可能已过时，询问是否归档
2. 同类问题合并 → 多个相似问题合并为一个条目
3. 答案已变化的问题 → 更新答案或标记"需验证"
```

### 技能维护（每周）

检查 SKILL.md → 合并重复 → 补充模板 → 验证路径

### 新技能沉淀

重复 2 次以上 → 创建 SKILL.md → 注册到技能列表

## 十三、自进化能力（Agent 自我成长）

### 能力一：自动沉淀技能改进

**触发信号**（用户说以下内容时自动触发）：
- "格式有问题..."、"下次要..."、"以后记住..."
- "不要..."、"改成..."、"换成..."
- 用户纠正当前输出格式/流程/规范
- 用户给出明确的新标准或模板

**执行流程**：
```
用户提出新要求 → 识别为"技能改进信号" → 确认相关技能 
→ 读取 SKILL.md → 找到修改位置 → edit_file 更新 
→ 告知用户："已沉淀到 {技能名} 技能文档 ✅"
```

**铁律**：
- **不要等用户说"更新技能"才动手**
- **用户的纠正/要求 = 立即触发技能更新**
- **更新后必须告知用户，让用户确认**

---

### 能力二：自动创建新技能

**触发信号**：
- 用户首次要求某类任务（且判断将来可能重复）
- 没有现成技能可以处理
- 任务具有可复用的流程/模板/规范

**判断标准**（将来可能重复）：
- 用户工作场景中会经常遇到
- 任务有明确的输入/输出/流程
- 值得固化成可复用的能力

**命名原则**：
| 类型 | 示例 | 说明 |
|------|------|------|
| ✅ 好 | 搬题姬、墨池、雀影 | 意象化、拟人化、2-4字 |
| ✅ 好 | 破晓、格知、见自己 | 有意境、朗朗上口 |
| ❌ 差 | XXX工具、XXX助手、XXX生成器 | 功能性、无辨识度 |
| ❌ 差 | 超长名字、英文混杂 | 难记忆、难传播 |

**新技能目录结构**：
```
skills/{技能名}/
├── SKILL.md          # 技能定义（必须）
├── modules/          # 子模块（可选）
├── templates/        # 模板文件（可选）
└── examples/         # 示例文件（可选）
```

**SKILL.md 必须包含**：
```yaml
---
name: {技能名}
version: 1.0.0
description: 一句话描述
allowed-tools: [Read, Write, Edit, ...]
metadata:
  trigger: 触发场景描述
---

# {技能名} - 功能描述

## 触发条件
## 执行流程
## 输出格式
## 示例
```

**执行流程**：
```
识别可复用任务 → 确认无现成技能 → 构思简洁名字 
→ 创建目录结构 → 写入 SKILL.md → 注册到技能列表 
→ 告知用户："已创建 {技能名} 技能 ✅"
```

**铁律**：
- **不要等用户说"创建技能"才动手**
- **识别到可复用模式 = 立即创建技能**
- **名字要简洁优雅，让用户愿意使用**

### 能力三：技能版本管理

**版本号规则**：`主版本.功能版本.修订版本`

| 变化类型 | 版本变化 | 示例 |
|---------|---------|------|
| 流程重构 | 主版本 +1 | 1.0 → 2.0 |
| 新增功能 | 功能版本 +1 | 1.0 → 1.1 |
| 格式/规范调整 | 修订版本 +1 | 1.0 → 1.0.1 |

**变更记录格式**：
```markdown
**更新日期**: YYYY-MM-DD

**变更记录**：
- YYYY-MM-DD: 变更内容简述
- YYYY-MM-DD: 初始版本
```
